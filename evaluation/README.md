# MediQ Evaluation Framework

Automated AI benchmarking and regression testing subsystem for the MediQ Healthcare Intake & Triage Platform.

> This subsystem is **completely independent** of the production workflow. It calls the triage engine directly without any database or HTTP server.

---

## Overview

The evaluation framework measures MediQ's AI quality across six tasks:

| Task | Metric |
|---|---|
| Symptom extraction | Precision / Recall / F1 |
| Clinical entity extraction | Per-entity accuracy |
| Risk classification | Accuracy + confusion matrix |
| Department routing | Partial-credit accuracy |
| Conversation quality | Repeat rate, context retention |
| Summary generation | Keyword coverage |

### Case Schema

Each evaluation case follows this structure:

```json
{
  "id": "TE_001",
  "source": "hand_curated",
  "original_record_id": null,
  "category": "Cardiology / Emergency",
  "patient_input": "I have chest pain that radiates to my left arm...",
  "expected_symptoms": ["chest pain"],
  "expected_entities": {
    "bodyPart": "chest, left arm",
    "severity": "9/10",
    "duration": "20 minutes",
    "mechanismOfInjury": null,
    "functionalLimitation": null,
    "associatedSymptoms": ["radiation to arm"],
    "medicalHistory": [],
    "medications": []
  },
  "expected_risk": "critical",
  "expected_department": "Emergency — Cardiology",
  "expected_summary_keywords": ["chest pain", "arm radiation", "severe", "20 minutes"]
}
```

### Source Values

| Value | Meaning |
|---|---|
| `provided_dataset` | Ingested from supplied medical dialogue dataset |
| `hand_curated` | Manually designed by clinical scenario author |
| `llm_paraphrase` | Generated by LLM from a parent case |

## Evaluation Methodology

### How Cases Are Evaluated

For each benchmark case:

1. The `patient_input` is fed into `processMessage()` directly (no HTTP, no DB)
2. The engine responds with a question
3. Generic simulated answers are provided (up to 6 turns)
4. After the session ends (or after 6 turns), predictions are collected
5. Predictions are compared against expected labels

### Metrics Definitions

**Symptom Extraction F1**
- Precision = correct extracted / total extracted
- Recall = correct extracted / total expected
- F1 = harmonic mean of precision and recall
- Fuzzy matching: a prediction is "correct" if it contains any word from the expected label

**Clinical Entity Accuracy**
- Per-field exact/fuzzy match across: `bodyPart`, `severity`, `duration`, `mechanismOfInjury`, `functionalLimitation`
- Fields not annotated in the ground truth are excluded from scoring

**Risk Classification Accuracy**
- Exact match of risk level: `low`, `medium`, `high`, `critical`
- Confusion matrix shows where misclassifications occur

**Department Routing Accuracy**
- Partial credit: 1.0 for exact match, partial for correct specialty area
- e.g. predicted "Emergency — Cardiology" vs expected "Emergency — Cardiology" → 1.0
- e.g. predicted "General Practice" vs expected "Emergency — Cardiology" → ~0.0

**Repeat Question Rate**
- `repeated_questions / questions_asked` per session
- Target: < 5%

**Context Retention Rate**
- Proportion of sessions where the AI did not re-ask for information already in the initial message
- Target: > 90%

**Keyword Coverage**
- Proportion of `expected_summary_keywords` found in the generated summary
- Only applicable when a summary is generated (completed sessions)

## Adding New Evaluation Cases

### To add a public dialogue case

Add to `datasets/public_dialogue_cases.json` following the schema above. Set `source: "provided_dataset"`.

### To add a hand-curated case

Add to `datasets/trauma_emergency_cases.json`. Set `source: "hand_curated"`.

### To add paraphrases

Either:
1. Manually add to `datasets/paraphrased_cases.json` with `source: "llm_paraphrase"` and `parent_id`
2. Run `npm run generate-paraphrases` (requires LLM configured in `server/.env`)

After adding cases, always rebuild:

```bash
npm run build-dataset && npm run validate-dataset
```

## Generating Paraphrases with LLM

Configure an LLM provider in `server/.env`:

```env
# OpenAI
OPENAI_API_KEY=sk-your-key-here

# OR Gemini
GEMINI_API_KEY=AIzaSy...

# OR Ollama (local)
OLLAMA_URL=http://localhost:11434
OLLAMA_MODEL=llama3
```

Then run:

```bash
# Generate 5 paraphrases from hand_curated cases
npm run generate-paraphrases

# Generate 20 paraphrases
node scripts/generate_paraphrases.js --count 20

# Generate from public_dialogue cases
node scripts/generate_paraphrases.js --source provided_dataset
```

## Running a Subset

```bash
# Evaluate only the first 20 cases
node scripts/evaluate.js --limit 20

# Evaluate only trauma/emergency cases
node scripts/evaluate.js --source hand_curated

# Evaluate only paraphrased cases
node scripts/evaluate.js --source llm_paraphrase
```

## Interpreting Results

After running `npm run evaluate`, check `reports/evaluation_results.json` for:

- `summary.metrics` — aggregated metrics
- `results[].transcript` — full conversation for each case
- `results[].risk_correct` — boolean pass/fail per case

After running `npm run generate-report`, check `reports/metrics_report.md` for:

- Publication-ready tables
- Per-category breakdown
- Failure analysis with sample cases
- Recommendations

## Regression Testing

Run the full pipeline after any change to:
- `server/services/triageEngine.js`
- `server/services/symptomExtractor.js`
- `server/services/reasoning/clinicalReasoner.js`
- Any prompt file in `server/prompts/`

```bash
cd evaluation && npm run full-pipeline
```

Compare the new `metrics_report.md` against the previous one to detect regressions.

---

## Architecture Notes

- The evaluation framework imports the server's triage engine **directly** — no HTTP calls, no MongoDB required
- `dotenv` is loaded from `server/.env` to pick up any configured LLM keys
- Sessions are simulated with a plain JavaScript object (no Mongoose) using the same schema fields the engine expects
- The `toObject()` method is mocked to simulate Mongoose serialisation
